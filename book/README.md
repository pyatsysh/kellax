# kellax by solved problems

This is a short book that teaches numerical continuation through a sequence of
nine worked problems, solved with [kellax](../README.md). In each chapter we
pose one equation, trace the branch of its solutions as the parameter varies,
and let the theory emerge from the problem at hand. The runnable script behind
each chapter can be found in [`examples/`](../examples), and the figure shown
in the chapter is regenerated by running that script. The problems range from
the cubic normal form of the fold, through pattern formation in the
Swift–Hohenberg equation, to the ignition of an exothermic reactor. Thus, the
book is a collection of solved problems, and not a course of lectures on the
abstract machinery of continuation.

## Principles

The first principle is that theory comes from problems. No chapter opens with
abstract machinery. Instead, a chapter opens with an equation whose solution
branch we want to compute, and the method appears exactly at the point where
the naive approach breaks down. In our experience this order is the natural one
for the subject: a reader who has watched parameter-stepping fail at a fold
does not need to be persuaded of the value of a bordered system.

The second principle is that every number is validated. For each problem we
state the known answer, given either by an analytic fold location or by a value
from the literature, and the script checks the output of kellax against it.
The printed numbers in each chapter are real output of the scripts. Nothing is
quoted that the reader cannot regenerate at the command line. Where a chapter
depends on a reference value, the value is stated in the text, and the check is
printed when the script runs.

The third principle is that the code is the source of truth. Inside the
chapters we show short excerpts only, so as to keep the prose readable. The
full, runnable version of each problem lives in `examples/<name>.py` and writes
its figure to `figures/<name>.png`. Everything can be regenerated by the
commands under *Running* below.

Finally, the book is matrix-free where it matters. Small ODE and normal-form
problems use the dense engine, which forms the Jacobian of the residual and
factorises it. Discretised PDE fields use the matrix-free engine, which solves
every bordered system by preconditioned GMRES on Jacobian–vector products and
never forms a matrix at all. The choice of the engine is stated for each
problem in the table of chapters below.

## What each problem teaches

In what follows, we summarise the lesson of each problem in turn. The first
five lessons are methodological, and the last four are lessons in application.

The fold is the reason for the existence of continuation software. Chapter 1
shows why the naive stepping of the parameter fails at a turning point: the
Jacobian of the residual becomes singular there, and just past the fold the
nearby solution ceases to exist. The bordered pseudo-arclength system, which
reparametrises the branch by its own arclength, walks through the turning
point without incident. The Moore–Spence refinement then pins the location
of the fold to Newton precision. Moreover, folds are generic, so failures of
this kind are common in practice.

A fold is codimension-1. Therefore, in two parameters it is not a point but a
curve, and in chapter 2 we trace such a curve with `track_fold`. Notice that a
single continuation recovers the whole of the fold set, with both arms
obtained at once, because the arclength parametrisation passes smoothly
through the cusp.

Chapter 3 brings the same machinery to a discretised boundary-value problem,
namely the Bratu–Gelfand equation, which AUTO, MatCont and BifurcationKit all
ship as a benchmark. Since every Jacobian is obtained by automatic
differentiation, the discretisation of the equation is the only modelling we
do, and the stability of the cool and the hot branches can be read off the
spectrum of the Jacobian.

In chapter 4 we run the same trace matrix-free, with GMRES acting on
Jacobian–vector products, and the method therefore keeps working when the
Jacobian can no longer be stored in memory. Chapter 5 takes that engine to a
pattern-forming PDE. The localized states of the Swift–Hohenberg equation live
on one connected branch, which wiggles through 38 folds as the patterns grow.
Indeed, the snake is the point of the exercise: a method of continuation
either survives a branch of this kind or it does not.

The remaining chapters are applications of the same tools. Chapter 6 applies
the methods of the first two chapters to the exothermic stirred-tank reactor,
whose ignition and extinction folds bound a hysteresis loop. Tracking the
folds in a second parameter then leads to the cusp where the hysteresis is
born. Chapter 7 moves the story to ecology and locates a pair of folds of the
Rosenzweig–MacArthur predator–prey model. In chapter 8 we return to ignition
on the unit square, where the fold now belongs to a 2-D thermal field.
Finally, chapter 9 differentiates the converged Moore–Spence system with
respect to the parameters of the model, and one linear solve then yields the
exact gradient of the ignition fold of the CSTR.

## Running

The commands below install kellax into a fresh virtual environment and
regenerate the figures of the book:

```bash
uv venv .venv && uv pip install --python .venv/bin/python -e .
uv pip install --python .venv/bin/python matplotlib
python examples/cubic_fold.py         # -> figures/cubic_fold.png  (+ validation to stdout)
python examples/cusp.py               # -> figures/cusp.png
python examples/bratu.py              # -> figures/bratu.png
python examples/bratu_matrixfree.py   # -> figures/bratu_matrixfree.png
python examples/swift_hohenberg.py    # -> figures/swift_hohenberg.png
python examples/cstr.py               # -> figures/cstr.png
python examples/predator_prey.py      # -> figures/predator_prey.png
python examples/bratu_2d.py           # -> figures/bratu_2d.png
```

The block regenerates the 8 figures of chapters 1–8. The figure of chapter 9
is regenerated by [`cstr_sensitivity.py`](../examples/cstr_sensitivity.py) in
the same way. See also the narrative [TUTORIAL](../TUTORIAL.md), which offers
a faster, API-first tour of the same calls.
